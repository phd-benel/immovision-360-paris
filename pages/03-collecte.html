🛠️ TP Phase 1 : Ingénierie du Data Lake "ImmoVision 360"

Fini la théorie, place à l'ingénierie. Vous êtes désormais responsables de la création de la zone "Bronze" (Raw Data) de notre projet sur la gentrification du Marais. Votre objectif n'est pas encore d'analyser, mais de collecter, stocker et certifier la matière première de manière reproductible et professionnelle.

Voici votre cahier des charges strict.

1. Les Sources de Données (Inputs)

Vous allez amorcer votre Data Lake à partir de sources issues de portails Open Data. Pour ce projet, nous utilisons les extractions officielles du portail Inside Airbnb (https://insideairbnb.com/fr/). Ce site milite pour la transparence des données de plateforme. Vous devez télécharger les fichiers correspondant à la ville de Paris (dernière mise à jour disponible) https://insideairbnb.com/fr/paris/ . Vous y trouverez initialement deux fichiers "mères" :

  • listings.csv https://data.insideairbnb.com/france/ile-de-france/paris/2025-09-12/visualisations/listings.csv : Contient toutes les annonces (ID de l'annonce, nom de l'hôte, prix, coordonnées GPS, et surtout la colonne picture_url qui contient le lien web vers la photo haute résolution de l'appartement).
  • reviews.csv https://data.insideairbnb.com/france/ile-de-france/paris/2025-09-12/visualisations/reviews.csv : Contient l'historique textuel des commentaires laissés par les voyageurs (ID de l'annonce, date, texte brut).

2. Architecture du Data Lake (La Rigueur Spatiale)

Un Data Lake n'est pas un dossier "Fourre-tout" sur votre bureau. Vous devez scripter la création d'une arborescence stricte et standardisée. Votre premier livrable doit respecter exactement cette hiérarchie physique : 

/ImmoVision360_DataLake
├── /data
│   └── /raw
│       ├── /tabular
│       │   ├── listings.csv
│       │   └── reviews.csv
│       └── /images
│           └── (Dossier initialement vide, à remplir via script)
├── /scripts
│   ├── 01_ingestion_images.py
│   └── 02_sanity_check.py
└── README.md

Règle d'or : Vos scripts Python ne doivent jamais contenir de chemins absolus (ex: C:/Users/MonNom/Desktop/...), mais uniquement des chemins relatifs (ex: ../data/raw/...) pour que le projet soit exécutable sur la machine de n'importe quel autre ingénieur.

3. Processus d'Ingestion & Scraping (Script 1)

Le fichier listings.csv contient des liens vers des images, mais le Data Lake a besoin des fichiers physiques (les pixels) pour que la Phase 2 (IA Vision) puisse fonctionner. Vous devez concevoir un script (01_ingestion_images.py) qui va agir comme un aspirateur intelligent.

La logique algorithmique exigée pour ce script :

  1. Lecture du catalogue : Le script doit ouvrir et lire le fichier listings.csv.
  2. Extraction des cibles : Pour chaque ligne du fichier, le script doit isoler deux informations vitales :
    • La colonne ID (l'identifiant unique de l'appartement, ex: 785412).
    • La colonne picture_url (le lien HTTP de la photo).
  3. Le Scraping (Téléchargement) : Le script doit se connecter à l'URL via une requête réseau, télécharger le fichier image, et le sauvegarder dans le dossier /data/raw/images/.
  4. La Règle de Nommage Strict : Le fichier image sauvegardé DOIT obligatoirement être nommé selon son ID. Si l'ID est 785412, l'image doit s'appeler 785412.jpg. C'est ce qui permettra de faire la jointure plus tard.
  5. Le Pilier d'Idempotence (Obligatoire) : Avant de télécharger une image, votre script doit vérifier si ID.jpg existe déjà dans le dossier /images/. Si oui, il passe à la suivante sans rien télécharger. Cela vous permet de couper le script à 50% et de le relancer sans écraser ce qui a déjà été fait.
  6. Gestion des Exceptions (Try/Except) : L'internet est instable. Des liens seront morts (Erreur 404), des requêtes vont expirer (Timeout). Votre script ne doit pas planter. Il doit intercepter l'erreur, l'afficher dans la console ("Lien mort pour l'ID XYZ"), et continuer son travail jusqu'à la dernière ligne.

Remarque : L'Audit Éthique et Légal (Le "Serment" de l'Ingénieur)

Un Data Scientist/Engineer n'est pas un pirate informatique. Avant d'écrire la moindre ligne de code pour aspirer des milliers de fichiers, vous devez vous assurer que vous en avez le droit. "Scraper" (extraire) de la donnée sur le web engage votre responsabilité éthique et légale.

Avant de lancer votre "aspirateur à images", votre script et votre démarche doivent impérativement respecter les 4 Règles d'Or de la Courtoisie Numérique :

  • La Règle du robots.txt (La Porte d'Entrée) :
    • Le Concept : Chaque site web sérieux possède un fichier à la racine de son serveur (ex: www.site.com/robots.txt) qui dicte aux machines ce qu'elles ont le droit de regarder ou non.
    • Votre Mission : Avant de cibler les URLs des images contenues dans votre CSV, vous devez vérifier mentalement (ou via une brève recherche) si les serveurs d'hébergement autorisent les bots automatisés sur ces chemins spécifiques.
  • La Courtoisie Serveur (Le "Rate Limiting") :
    • Le Concept : Si votre script demande 5 000 images en 2 secondes, le serveur d'en face va considérer cela comme une attaque informatique (DDoS) et va planter, ou bloquer définitivement votre adresse IP (Erreur 429 : Too Many Requests).
    • Votre Mission : Votre script Python doit inclure une pause artificielle (un sleep ou délai d'attente) d'une fraction de seconde à quelques secondes entre chaque téléchargement d'image. Oui, votre script mettra plus de temps à tourner, mais c'est la marque d'un code professionnel et respectueux de l'infrastructure d'autrui.
  • L'Identification (Le "User-Agent") :
    • Le Concept : Quand vous naviguez sur le web, votre navigateur dit au serveur : "Bonjour, je suis un humain sur Google Chrome". Quand un script Python se connecte, il dit par défaut : "Bonjour, je suis un script Python". Beaucoup de serveurs bloquent automatiquement les scripts anonymes pour se protéger.
    • Votre Mission : Renseignez-vous sur la notion de Headers et de User-Agent dans vos requêtes réseau. Soyez transparents sans pour autant vous faire bloquer bêtement par un pare-feu basique.
  • L'Analyse des Licences et du RGPD (La Loi) :
    • Le Concept : Les fichiers listings.csv et reviews.csv proviennent de l'Open Data. Mais "Open" ne veut pas dire "sans règles".
    • Votre Mission : Vérifier la licence d'utilisation (Est-ce une licence Creative Commons ? Avez-vous le droit de l'utiliser pour un projet commercial ou uniquement académique ?). Veuillez lire : https://insideairbnb.com/about/ + https://insideairbnb.com/fr/data-policies/ + https://insideairbnb.com/fr/get-the-data/

4. Le Bilan de Santé de la Donnée (Script 2)

Faire confiance à un téléchargement est une erreur de débutant. Une fois le script 1 terminé, vous devez auditer votre Data Lake. Vous allez coder un second script (02_sanity_check.py) qui fera le contrôle qualité.

La logique algorithmique exigée pour le Sanity Check :

  1. Comptage Théorique : Le script lit listings.csv et compte le nombre exact de lignes (ex: 5000 annonces). C'est le nombre d'images "attendues".
  2. Comptage Physique : Le script compte le nombre de fichiers .jpg réellement présents dans /data/raw/images/ (ex: 4850 images).
  3. La Jointure Physique (Le Test Ultime) : Le script parcourt la liste des IDs du CSV un par un. Pour chaque ID, il vérifie physiquement sur le disque dur si le fichier <ID>.jpg est là.
  4. Génération du Rapport : Le script doit imprimer dans le terminal un rapport clair :
    • Total annonces CSV : X
    • Total images téléchargées : Y
    • Taux de complétion : Z %
    • Liste des "Orphelins" : (Afficher les 5 premiers IDs dont l'image manque à l'appel).

5. Le Livrable Final

Le code seul ne vaut rien s'il n'est pas documenté. Vous devez rédiger un fichier README.md à la racine de votre dossier. Ce fichier est votre rapport de livraison à destination de l'Architecte de Données.

Le README.md doit obligatoirement contenir ces sections :

  • Titre et Contexte : Une brève description de la mission (Création du Data Lake pour ImmoVision 360).
  • Structure du Répertoire : Reproduisez l'arborescence physique de vos dossiers pour expliquer comment vous avez rangé la donnée.
  • Notice d'Exécution : Expliquez pas à pas comment un autre ingénieur doit lancer vos scripts (ex: "1. Installez telle librairie. 2. Lancez le script 1. 3. Lancez le script 2").
  • Audit des Données (Résultats du Sanity Check) : Vous devez écrire noir sur blanc les résultats de votre script 2. Je veux y lire le nombre exact d'images que vous avez réussi à ingérer, le nombre d'erreurs rencontrées, et votre taux de réussite.
  • Analyse des Pertes : Si vous avez 5000 lignes dans le CSV mais seulement 4800 images, rédigez un paragraphe expliquant techniquement pourquoi il y a une déperdition (liens morts expirés par Airbnb, blocage anti-bot, erreurs serveurs, etc.). C'est ici que l'on reconnaît un ingénieur d'un simple codeur.

Rendu attendu : Un lien vers un dépôt Git contenant l'architecture exacte, vos deux scripts .py opérationnels, et votre README.md complété.

(Note importante : Le dossier images doit rester vide sur GitHub pour éviter un repository trop lourd et saturer les serveurs. Vous conservez bien sûr le dossier rempli sur votre PC local pour vos tests, mais pour empêcher Git de l'envoyer en ligne, vous devez utiliser un fichier .gitignore.)

Le .gitignore est un fichier texte placé à la racine de votre projet. Il indique à Git quels fichiers ou dossiers il doit ignorer (ne pas suivre et ne pas uploader). C'est la barrière de sécurité qui sépare votre code (léger) de vos données brutes (lourdes).

Créez un fichier nommé exactement .gitignore (attention au point au début) et écrivez simplement cette ligne à l'intérieur : 

data/raw/images/

En faisant cela, Git "verra" le dossier sur votre ordinateur, mais il fera comme s'il n'existait pas lors du git commit et du git push.
